# Prod release notes via GitHub Releases

**Date:** 2026-07-21
**Status:** Approved — ready for implementation plan
**Scope:** `.github/workflows/deploy-prod.yml` (one new job)

## Problem

The "Deploy Production" workflow is the manual, windowed prod release. It already
has a `summary` job that writes a per-target pass/fail table and the live URLs to
the run's `$GITHUB_STEP_SUMMARY`. But that summary:

- says nothing about **what code** is shipping (no PR/commit list), and
- is **ephemeral** — it lives only inside that Actions run, so there is no durable,
  browsable record of what each production release contained.

We want (1) a summary of the changes included in each prod release, and (2) a
durable record of those release notes kept in the GitHub repo.

## Decisions (locked)

- **Durable store:** GitHub Releases + an annotated tag per deploy. Lives in the
  repo's **Releases** tab — permanent, browsable, tied to the deployed commit. No
  commit-back to the source tree.
- **Notes content:** the list of **PRs merged since the previous prod release**,
  auto-generated by GitHub (`--generate-notes`).
- **When:** **always**, on every run — the deployed commit becomes prod the moment
  `advance-prod` fast-forwards the `prod` pointer, so we tag it regardless of
  individual target outcomes. Gated only on `advance-prod` succeeding (if the guard
  blocks or the fast-forward fails, nothing became prod → no release).

## Design

Add a single `release-notes` job to `deploy-prod.yml`.

### Baseline computation — none needed

GitHub's generated release notes use the **previous release tag** as the baseline
automatically. Each new release diffs against the last one, so we never capture the
old `prod` ref before `advance-prod` moves it. Because prod ships the exact
`staging` commit and its merge commits are reachable from that commit, the
tag-to-tag PR list is accurate.

First release has no prior tag, so its notes will be sparse (from repo start) — a
one-time effect.

### Tag naming

`prod-YYYY-MM-DD-<short-sha>` — e.g. `prod-2026-07-21-711371a`.

- Date for human scanning.
- Short-sha suffix so multiple deploys on the same day never collide.

Release is marked `--latest`.

### The job

```
release-notes:
  name: Tag + release notes
  needs: [advance-prod, deploy-api, deploy-chat-ingest,
          amplify-forms, amplify-landing, amplify-chat, amplify-feature-flagging]
  if: always() && needs.advance-prod.result == 'success'
  runs-on: ubuntu-latest
  permissions:
    contents: write   # create tag + release; job-scoped, deploy jobs stay read-only
  steps:
    - checkout at github.sha (fetch-depth: 0 so the tag lands on the right commit)
    - create tag + release:
        TAG="prod-$(date -u +%Y-%m-%d)-$(git rev-parse --short HEAD)"
        gh release create "$TAG" \
          --target "${{ github.sha }}" \
          --title "Production release $TAG" \
          --generate-notes \
          --latest
    - append the generated notes body to $GITHUB_STEP_SUMMARY
        (gh release view "$TAG" --json body -q .body >> "$GITHUB_STEP_SUMMARY")
```

- `GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}` for the `gh` calls.
- No AWS role assumed — this job only talks to the GitHub API.
- `date -u` for a stable UTC date regardless of runner locale.

### Interaction with the existing `summary` job

`summary` is left as-is (pass/fail table + URLs). `release-notes` is additive: it
produces the durable Release and echoes the PR list into the run summary. The two
jobs are independent.

## Out of scope (YAGNI)

- Committing a `CHANGELOG.md` / `docs/releases/` file back to the repo (chosen
  against — Releases tab is the store).
- Semantic version numbers (no versioning scheme exists; date+sha tags suffice).
- Changing sandbox/staging deploy workflows — this is prod-only for now.
- Failing the deploy if release creation fails — record-keeping must never gate the
  rollout; the step should be best-effort/non-blocking.

## Verification

- Trigger a `workflow_dispatch` prod deploy (or dry-run in a fork/branch): a
  `prod-YYYY-MM-DD-<sha>` tag and a matching GitHub Release appear, marked latest,
  with a PR list body.
- The Actions run summary shows the same PR list beneath the existing target table.
- A second deploy produces a new release whose notes cover only PRs merged since the
  first.
- zizmor passes (job-scoped `contents: write`, no floating action pins added).
