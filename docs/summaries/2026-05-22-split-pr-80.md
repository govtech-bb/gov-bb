# 2026-05-22 — Split PR #80 into four focused PRs

## Context

PR [#80](https://github.com/govtech-bb/gov-bb/pull/80) ("Content fixes: chat chrome, landing nav, forms layout, Start CTAs") bundled four unrelated concerns into one 29-file, 8-commit change. Reviewing it as one unit was awkward — the chat-chrome work and the Start-CTAs regression fix have nothing in common except they were both authored in the same sitting.

We chose to split rather than just-merge-and-move-on because one of the four (Start CTAs) is a user-visible regression worth landing on its own merits, and the others have independent reviewers / risk profiles.

## What we did

Cut four branches from `origin/dev`, cherry-picked the original commits per concern, opened PRs, closed #80 with a link comment.

|---|---|---|
| [#90](https://github.com/govtech-bb/gov-bb/pull/90) | `feat/chat-gov-chrome` | `1d2d2e7`, `c9b90a1`, `2413908`, plus a regenerated lockfile commit |

## Why this shape

- **Parallel branches, not stacked.** Verified the four concerns touch disjoint files (chat/* vs forms/* vs landing/Header.tsx vs landing/MarkdownContent + content/*). No cross-PR dependency means stacked review machinery would have been pure overhead.
- **Cherry-pick, not squash.** Preserves Harry's authorship and the per-concern commit messages — `git blame` and review history stay legible. Squashing would have invented four new commits with new SHAs and lost the granular `feat(chat): compress chrome and pin composer on mobile` vs `feat(chat): wrap app in Government of Barbados header and footer` distinction.
- **Lockfile regenerated, not cherry-picked.** The original lockfile commit (`6d190cb`) was a snapshot against a `dev` that had since moved — PR #86 (form_builder reorg) landed in between, shifting AI deps between workspaces. Cherry-picking conflicted on `pnpm-lock.yaml`. Skipped the original commit and re-ran `pnpm install --lockfile-only` against the current `dev`. Net change in PR #90: 3 lines (just chat's importer entry for `@govtech-bb/react`, which is a workspace package and already in the lockfile).
- **PR #80 closed, not converted.** Force-pushing a trimmed `content-fixes` branch to re-target it as one of the four would have churned the review history. Opening four fresh PRs and closing #80 with a comment was cleaner.

## Verification

File-level diff: union of files touched across the four split branches equals the file set in `content-fixes` exactly (modulo lockfile, which differs by design).

Did not re-run UI verification locally — the splits carry the same source commits forward; CI on each PR is the new verification surface.
