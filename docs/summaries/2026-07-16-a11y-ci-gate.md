# Session summary — Accessibility (axe) CI gate (#1971)

**Date:** 2026-07-16 · **Branch:** `ci-1971-a11y-axe-gate` (off `main`)

## What shipped

An automated accessibility gate: on every previewed PR, a Playwright job opens
the deployed **forms** preview in real Chromium and runs axe-core, failing the
PR on any **serious/critical** WCAG 2.0/2.1 A-or-AA violation.

- `apps/forms/playwright.a11y.config.ts` — deployed-URL scan config (no submit).
- `apps/forms/e2e/a11y/forms.a11y.spec.ts` — scans a public form's first step +
  the not-found page.
- `apps/forms/e2e/helpers/a11y.ts` — the axe runner + serious/critical filter.
- `.github/workflows/forms-a11y.yml` — reusable workflow (mirrors forms-smoke).
- Wired into `pr-preview.yml` as the `a11y-forms-preview` job.
- `@axe-core/playwright` dep + `test:a11y` script; `e2e/a11y` excluded from the
  normal suite; `test-results/` git-ignored.
- Docs: `docs/accessibility-ci-gate.md`.

## Why it looks the way it does

- **Real browser, not jsdom.** The existing `jest-axe` unit tests run in jsdom,
  which has no CSS/layout — so they structurally cannot catch colour-contrast,
  focus, or focus-ring issues, several of which were among the ~15 a11y bugs the
  team had already fixed by hand. A live-browser axe scan is the only thing that
  guards those. This is the core reason the gate exists at all rather than "just
  add more unit tests."

- **Reuses the smoke infrastructure, doesn't reinvent.** forms already ships
  `@playwright/test` and a live-URL harness (`playwright.smoke.config.ts` +
  `forms-smoke.yml` called from `pr-preview.yml`). The a11y gate mirrors that
  pattern exactly, so it's familiar to reviewers and rides existing preview URLs.

- **Separate from the smoke, on purpose.** The smoke test *submits a form for
  real*; the a11y scan only *loads and scans*. Keeping them separate means the
  a11y job needs no submission token, creates no submission rows, and (unlike the
  smoke) has no form-contract-skew dependency — so it can run on every previewed
  PR including prod-targeted ones.

- **Blocking, but only serious/critical WCAG A/AA.** Chosen over "advisory" only
  after confirming the scanned pages already pass on sandbox, so it won't
  red-wall unrelated PRs. Moderate/minor and axe *best-practice* rules (e.g.
  `heading-order`) are intentionally out of scope for now — the gate lands green
  and the threshold can tighten later. This keeps the check trusted rather than
  routinely overridden.

- **Scan targets = shared UI, kept robust.** The first slice scans the first step
  of `jobstart-plus-programme` (the same public, upload-free form the preview
  smoke uses — must be public since the frontend reads the sandbox API, which
  404s non-public forms) and the not-found page (shared `ErrorPage`). These
  exercise the shared field/label/heading/landmark/focus/contrast surface without
  form-specific field-filling. Deeper steps and landing/chat coverage are
  deliberate follow-ups.

## Verification

Real Chromium against sandbox: both pages 0 serious/critical → pass. A
throwaway negative-control injecting an unlabeled input made the gate fail as
expected, proving the detect→report→fail path. Specs compile, are discovered by
their config, and are excluded from the normal e2e/unit run.

## Follow-ups

- Extend the same live-URL axe pattern to **landing** and **chat** (previews
  already exist) — the issue's remaining acceptance criteria.
- Optionally tighten the threshold (include `moderate`) once baseline is clean.
