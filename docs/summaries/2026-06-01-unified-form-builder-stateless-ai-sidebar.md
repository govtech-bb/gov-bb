# form_builder: unify UI + AI into one screen with a stateless AI sidebar

## Context

Implemented from `docs/plans/490-unified-form-builder.md` on branch
`feat/490-unified-form-builder` (merges into `sandbox`). Issue
[#490](https://github.com/govtech-bb/gov-bb/issues/490).

The builder was two sibling surfaces behind a two-card landing: `/builder/ui`
(visual editor, client-side draft) and `/builder/ai` (a chat page backed by an
in-process `Map<id, Session>` that replayed a growing message history). The only
bridge was heavyweight — the AI page published its recipe to `form_definitions`,
then re-opened it in the UI builder by `?formId=` (ADR 0011). Two costs: the
session map was lost on restart and unshareable across instances (#332), and
editing an AI-generated form required a DB round trip even for an unsaved tweak.

The goal: **one `/builder` screen** — the visual editor with a collapsible AI
sidebar docked beside it, reading and writing the *same* live draft, no route
switch and no DB round trip.

## What we did

**Backend (`apps/form_builder_api`)**

- **`src/routes/ai.ts`** — replaced ~416 lines of session machinery (the
  `Map<id, Session>`, `/sessions` CRUD, `/message`, `/extract`, `/publish`,
  `/delete`, `/presigned-upload`) with a single stateless
  `POST /builder/ai/convert`: `{ message?, recipeJson?, pdfBase64? }` →
  `{ recipe, reply }`. `buildSystemPrompt` appends live custom components per
  call; `buildUserText` fences the incoming recipe JSON into a single user turn;
  the existing `extractRecipe` + Bedrock `chat()` PDF path are kept. `/status`
  stays.
- Deleted `src/ai/s3.ts` and `src/ai/sql-builder.ts` (only the removed
  presigned/session specs used them) and dropped `@aws-sdk/client-s3` +
  `@aws-sdk/s3-request-presigner` from `package.json`.
- New `ai.convert.spec.ts` (edit / upload / no-recipe / custom-components /
  503 / 400 / 500); removed `ai.message.spec.ts` and
  `ai.presigned-upload.spec.ts`.

**Frontend (`apps/form_builder`)**

- **Route move** — `app/routes/builder/ui/*` promoted to `app/routes/builder/*`
  (all `-`-prefixed co-located helpers + specs, and `index.tsx`), so `/builder`
  *is* the editor. Deleted the two-card landing and `builder/ai/index.tsx`;
  removed `/builder/ui` and `/builder/ai` entirely (no redirects). Route tree
  regenerated by the build. Import depth shifted `../../../` → `../../`.
- **New `-ai-sidebar.tsx`** — collapsible right-dock panel sharing the editor's
  `draft` / `version`. Two distinct actions: **Upload** (PDF/image → recipe,
  4 MB client guard, 413→"Invariant failed" decode) and **Edit Form** (text
  tweak; serializes the draft as `recipeJson`). Holds the transcript
  client-side.
- **Apply pipeline** — `applyAiRecipe` in `index.tsx`: deserialize → no-op guard
  → uniqueness pre-flight (`findRecipeIdCollisions`) → server `validateRecipe` →
  confirm-if-dirty → `LOAD_DRAFT` → `bumpPatch`. New `-apply-recipe.ts`
  (renamed from `-open-from-ai.ts`) keeps `buildLoadArgs` and adds `draftsEqual`.
- **Removed dead handoff** — the `?formId=`-from-AI `useEffect`, `parseBuilderSearch`,
  `validateSearch`, the StepList "Switch to AI" button, and `handleSwitchToAi`.
  Renamed `server/ai-builder/sessions.ts` → `convert.ts` (single
  `convertRecipe`); `types.ts` drops `SessionResponse`/`PublishResponse`, adds
  `ConvertRequest`/`ConvertResponse`.
- New `-ai-sidebar.spec.tsx` (edit-applies, conversational no-op, error
  surfacing, collapse) and `-apply-recipe.spec.tsx` (`buildLoadArgs`,
  `draftsEqual`).

**Docs** — rewrote the AI sections of `apps/form_builder/SPEC.md` for the
stateless/unified model; ADR 0022 records the principle and supersedes 0011.

## Why we did it that way

- **Backend composes the prompt; the client sends parts.** `convert` takes
  `message` + `recipeJson` as separate fields and fences them server-side,
  rather than the client pre-building a prompt string. Keeps the wire contract
  literal and the prompt shape in one place.
- **`convert` returns `{ recipe, reply }`, not just `{ recipe }`** (a small
  extension of the plan). The sidebar renders a conversation, so it needs the
  assistant text; and a conversational reply legitimately has `recipe: null` —
  the sidebar shows `reply` and leaves the draft untouched. This is how
  "a purely conversational prompt does not bump the version" falls out for free.
- **No-op guard runs before validate, not just before the bump.** The plan
  gated only the version bump on "recipe changed". We short-circuit earlier: an
  unchanged recipe skips the server validate, the confirm, and the load
  entirely. `draftsEqual` compares the two drafts *serialized with a fixed
  version and with `createdAt`/`updatedAt` stripped* — necessary because
  `serializeRecipeDraft` stamps timestamps and `deserializeRecipe` mints fresh
  editor-only field ids, so neither a raw-draft nor a raw-serialized comparison
  would hold.
- **Apply reuses the editor's own validation, in place.** `applyAiRecipe` runs
  the same collision pre-flight + `validateRecipe` the Save/Deploy buttons use,
  and only ever `LOAD_DRAFT`s on the valid+changed+confirmed path — so a broken
  AI recipe surfaces an error and never clobbers good work. It deliberately
  bumps a patch off the *working* version rather than adopting the recipe's own
  version (the AI's version field is not authoritative).
- **Old routes removed, not redirected** (user decision) — bookmarks to
  `/builder/ui` / `/builder/ai` now 404; the home route already redirects `/` →
  `/builder`.
- **S3 deps removed because the path was never live.** The presigned-upload flow
  was dormant scaffolding keyed per session; the session model was its only
  anchor, so it went with it. Upload is inline-base64 only. A future cap-lift
  would reintroduce a *stateless* presigned flow.
- **SPEC target corrected.** The plan named a non-existent
  `apps/form_builder_api/SPEC.md`; the AI-builder docs actually live in
  `apps/form_builder/SPEC.md`, which is what we updated.
