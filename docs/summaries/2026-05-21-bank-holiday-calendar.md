# Bank holiday calendar — landing migration

Ported the `/bank-holiday-calendar` page from the original NextJS
`frontend-alpha` project to `apps/landing` (Vite + TanStack Start).

## Shape

Split into two layers rather than porting as a single ~700-line route file:

- `apps/landing/src/lib/bank-holidays.ts` — pure date helpers (anonymous
  Gregorian Easter algorithm, `addDays`, `nthWeekdayOfMonth`,
  `startOfDay`, `daysBetween`, formatters) and `getBankHolidaysForYear`,
  which encodes the Public Holidays Act, Cap. 352 substitution rules.
- `apps/landing/src/lib/bank-holidays.test.ts` — 9 vitest cases
  pinning down Easter dates, Kadooment computation, and each of the
  four substitution-rule branches plus a no-substitutes year.
- `apps/landing/src/routes/bank-holiday-calendar.tsx` — UI only.

The branchy substitution rules (Monday-in-lieu for Jan 1, Jan 21, Apr 28,
May 1, Nov 30, Dec 26 on Sunday; Tuesday-in-lieu for 1 Aug on Sunday or
Monday; Tuesday-in-lieu for 25 Dec on Sunday) are exactly the kind of
logic that pays off being unit-tested. Keeping it in a non-React module
also means the route file stays focused on layout.

## Stack adaptations

- Dropped the source page's per-page `<StageBanner stage="alpha" />` —
  the landing root layout already renders one globally via `Header.tsx`,
  so the source's wrapper block would have produced a duplicate banner.
- Dropped `<Suspense>` — no async boundary needed in this stack.
- Replaced `useRouter` + `useSearchParams` with TanStack's
  `validateSearch: z.object({ year: z.coerce.number().int().min(...).max(...).optional() })`
  on the route, read via `Route.useSearch()`.

## Year switcher: Link, not useNavigate

The first cut used `useNavigate({ from: Route.fullPath })` and a
programmatic `navigate({ search, replace: true })` call from `onClick`.
Reload showed the buttons did nothing. After two rounds of guesswork
adjustments (drop `from`, add explicit `to`) didn't fix it, switched
to TanStack `<Link>` components — the same pattern every other route
in the landing app uses for navigation.

Lesson worth keeping: when a navigation API isn't working and there's
a declarative alternative the codebase already uses, swap rather than
debug. `<Link to=... search={{ year: n }} replace />` is type-checked,
hydration-stable, and avoids the `useNavigate` ambiguity entirely.

The disabled state at `MIN_YEAR` / `MAX_YEAR` is rendered as a
`<span aria-disabled>` so non-interactive years can't be clicked while
keeping the same visual style as the active button.

  on Vite's `import.meta.glob`). Pre-existing since the landing app
  was added; unrelated to this work.
- `src/content/mda.ts:64` — `ReferenceError: Buffer is not defined`
  from gray-matter being pulled into the browser bundle. Logs an
  unhandled rejection on every page load in dev. Pre-existing,
  unrelated, and the page renders despite it.
