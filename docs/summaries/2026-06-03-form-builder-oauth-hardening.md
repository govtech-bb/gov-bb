# form_builder OAuth security/UX hardening (#107)

Date: 2026-06-03
Issue: [#107](https://github.com/govtech-bb/gov-bb/issues/107) (parent: #30)
Branch: `worktree-form-builder-oauth-hardening-107` → `sandbox`
Plan: `docs/plans/107-form-builder-oauth-hardening.md`

## Why this work happened

PR 4 (Builder Hosting + GitHub OAuth) shipped with four code-review findings
deliberately deferred so the plan author could weigh them on their own merits.
#107 tracked them. The goal: shrink the OAuth token blast radius, close a
cross-origin logout DoS, and make failure paths degrade gracefully instead of
throwing 500s.

The issue text (2026-05-22) was partly stale — it cited `auth.ts` line numbers,
but the flow was refactored after filing into `app/routes/auth/*` route
`beforeLoad` handlers plus pure helpers in `app/server/`. The cookie writes live
inline in `beforeLoad` (not in `createServerFn`s) because `setResponseHeader`
inside an RPC handler targets the RPC response, not the route's navigation
response, so a subsequent `throw redirect` would drop the `Set-Cookie`.

## What changed

### Scope narrowed: `repo read:org read:user` → `public_repo read:org`

`auth/github.tsx`. `read:user` was unnecessary (`GET /user` returns `login`
without it). `repo` was narrowed to `public_repo` — gov-bb is public, so this
still covers the collaborator-permission read and the Contents-API recipe
writes. `read:org` stays (the team-membership check needs it). A CAVEAT comment
marks the one-way door: **if gov-bb is ever made private, `public_repo` silently
breaks the permission check and publish flow — bump back to `repo`.**

### GET `/auth/logout` deleted; replaced by a POST `logoutSession` RPC

The GET route was a CSRF/DoS vector: a cross-origin `<img src=".../auth/logout">`
logged the user out because `SameSite=Lax` sends the cookie on top-level
navigation. `logoutSession` is now a `createServerFn({ method: "POST" })` in
`auth.ts` — TanStack RPC's non-simple content-type can't be reproduced by an
`<img>` or simple cross-origin form, so the attack no longer fires. Unlike the
OAuth flows it clears the cookie via `setResponseHeader` on the RPC response
itself (no redirect thrown, so the same-origin fetch response carries the
`Set-Cookie`). The route file and all `routeTree.gen.ts` entries were removed.

### CSRF state mismatch → graceful redirect, not a 500

`auth/github_.callback.tsx`. The mismatch branch threw a raw `Error`
(serialized as a 500 with no recovery path). It now
`throw redirect({ to: "/auth/denied", search: { reason: "csrf" } })`.

### `denied.tsx` rewired and made reason-aware

Added `validateSearch` for optional `reason`. `reason === "csrf"` renders
"Sign-in link expired" copy + a "Sign in again" CTA; the default renders the
existing no-write-access message. The old `<Link to="/auth/logout">` became a
button that `await logoutSession()` then navigates.

### `normalizeOAuthBase` helper (TDD)

A trailing slash on `OAUTH_REDIRECT_BASE` (`https://x.com/`) produced a
doubled-slash `redirect_uri` that won't match the registered GitHub callback.
Added a crypto-free, client-safe `normalizeOAuthBase(raw)` to `session.ts`
(strips one trailing slash), applied at both read sites. Written test-first.

## Notable choices

- **Hard navigation on the denied button — deviation from the plan.** The plan
  said `router.navigate({ to: "/auth/github" })`. That would break the restart:
  `/auth/github`'s `beforeLoad` sets the CSRF state cookie via a server-only
  `setResponseHeader`, which **no-ops under client-side navigation**, so the
  cookie would never be written and the next callback would fail CSRF. Used
  `window.location.assign("/auth/github")` instead — a full-page navigation makes
  `beforeLoad` run on the server. This mirrors how the deleted `logout.tsx`
  relied on `redirect({ href })` for a hard nav.
- **`routeTree.gen.ts` hand-edited.** Normally regenerated by the TanStack vite
  plugin, but worktrees have no `node_modules`. The file is `// @ts-nocheck`, so
  only structural correctness matters; all eight `AuthLogout*` sites were removed
  by hand and verified by grep.
- **Repo-owner/name display constant.** Exported `REPO_OWNER`/`REPO_NAME` from
  `github-oauth.ts` for the denied page rather than hard-coding the string.
  Documented as display-only (runtime API calls scope via `GITHUB_ORG`). This
  leaves the owner/name defined in several places across the package — tracked
  as a follow-up to consolidate, deliberately out of scope here.

## Findings explicitly NOT actioned

- **Finding 5 (no spec for the OAuth routes):** the pure network helpers are
  already fully tested (`github-oauth.spec.ts`); route `beforeLoad` logic isn't
  unit-testable in the current harness (no route/component test setup). Accepted
  gap. Only the new pure helper got a unit test.
- **`?payload` code-leak note:** moot — the callback is a route reached via
  GitHub's server redirect, not a `createServerFn`, so the OAuth `code` never
  lands in a `?payload` URL. No code change.
- **`safeEqual` length short-circuit:** informational only. Left a comment; no
  behaviour change (fixed-length hex CSRF tokens make the timing unobservable).
