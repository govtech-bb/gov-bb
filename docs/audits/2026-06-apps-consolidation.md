# `apps/` Consolidation Audit

**Date:** 2026-06-16  ·  **Scope:** `apps/{api,chat,form_builder,form_builder_api,forms,landing}` (excludes `apps/cms`)  ·  **Tracking EPIC:** [#1423](https://github.com/govtech-bb/gov-bb/issues/1423)

## Executive Summary

Audit of the gov-bb pnpm+nx monorepo (apps/api, chat, form_builder, form_builder_api, forms, landing; cms excluded) consolidating 68 raw semantic findings from 6 analysts against fallow v2.97 static-analysis ground truth. fallow reports the codebase is broadly healthy (avg maintainability 92.3, avg cyclomatic 1.6, only 8.06% overall duplication, and just 9 of 179 clone families crossing app boundaries) — so the value here is NOT mass copy-paste removal but (a) a handful of confirmed cross-app clones (analytics, OfficialBanner, mda-contacts, recipe-loader↔github-recipes), (b) divergent TYPES/CONTRACTS that share a name but not a definition and are caught only by live smoke tests (FormDefinitionSummary, SubmissionValues, ApiResponse envelope, semver comparators, MdaContact), (c) shared-library opportunities where a published package already owns the primitive but apps reimplement it (@govtech-bb/react OfficialBanner + input set; @govtech-bb/ai-bedrock; @govtech-bb/form-validation date formatter), and (d) fallow-only architectural/hygiene debt (6 circular deps all hubbed on the forms form-builder barrel, unused files/deps, phantom undeclared deps, and ~180 over-complex functions). The single largest frontend duplication surface is apps/forms hand-rolling the government input components in raw govbb-* CSS instead of consuming @govtech-bb/react (which chat already does). The riskiest items are the unauthenticated apps/api admin endpoints and the four divergent semver comparators that drive 'latest recipe version' selection.

## Method

Two independent tracks were merged:

1. **Semantic multi-agent workflow** — 12 inventory agents (2 per app) mapped each app's surface; 6 cross-app analysts (one per objective: duplicated utilities, divergent types, API/integration layers, UI components, competing patterns, dead code) verified findings against the real code; a synthesis pass deduplicated and prioritized.
2. **`fallow` static analysis** (v2.97, scoped to `apps/*,!apps/cms`) — token/AST-level duplication, dead code, dependency hygiene, circular dependencies, and complexity. Used as ground truth to **confirm or downgrade** semantic claims and to **add** issues only a static analyzer can prove.

**Confidence:** ✅ *confirmed* = backed by fallow's static data (12 issues); ○ *likely* = evidenced semantically as a divergent type/pattern but **not** flagged as a token-level clone by fallow, so treat as structural rather than verbatim duplication (22 issues). No speculative issues survived.

#### `LIB-04` [#1392](https://github.com/govtech-bb/gov-bb/issues/1392) — AWS Secrets Manager fetch+cache wrapper hand-rolled in chat and form_builder (landing has a 3rd variant)

**Severity:** Medium · **Confidence:** ○ likely · **Apps:** apps/chat, apps/form_builder, apps/form_builder_api

**Problem.** The Amplify-Compute secret-resolution pattern (bake ARN at build, fetch value at runtime, lazy SecretsManagerClient singleton, Map<arn,Promise> in-flight cache, delete-on-failure so transient errors self-heal) is hand-rolled separately in chat and form_builder. The only real difference is chat caches the raw SecretString while form_builder JSON-parses it. apps/landing bakes secrets into Nitro runtimeConfig (a third variant of the same 'Amplify Lambda can't read runtime env' workaround, issue #202/#203), and form_builder_api resolves an admin token similarly.

**Evidence.** Both files declare `let client; const cache = new Map(); function getClient(){ if(!client) client = new SecretsManagerClient({}); return client; }` then a getter doing `getClient().send(new GetSecretValueCommand({ SecretId: arn })).then(r => { if(!r.SecretString) throw...; }).catch(err => { cache.delete(arn); throw err; })`. The in-flight-promise dedupe + delete-on-failure control flow is identical; only the post-fetch transform differs (string vs JSON.parse). The leading explanatory comments (citing #202) are near-identical.
